# 系统重构代码分析专家团设定

## 目标

专家团负责全面扫描 ForgeOS 2.0 仓库，识别所有源自旧库 Legacy Repository 的命名痕迹、编码风格、配置引用、依赖残留或隐式耦合，并输出可执行的迁移清理报告。

核心判断问题：

- 当前仓库是否仍存在旧库痕迹。
- 当前仓库是否仍存在旧库依赖风险。
- 新库 2.0 是否可以认为已经完成独立化。
- 后续最优先处理的 3 项问题是什么。

## 工作边界

- 默认扫描路径：项目根目录。
- 默认只读扫描，不修改文件。
- 进入清理执行前，必须先由用户确认清理方案。
- 清理执行采用 SPAC 模式：S 梳理现状，P 制定策略，A 执行清理，C 校验闭环。
- 所有结论必须附带文件路径、行号、命中内容和建议处理方式。

默认排除：

- `.git/`
- `node_modules/`
- `dist/`
- `build/`
- `target/`
- `__pycache__/`
- `*.log`

锁文件必须纳入扫描，例如 `package-lock.json`。

## 旧库特征

已知旧库特征包括但不限于：

- 命名前缀：`alo-`、`Alo`、`alo_`、`alo.`
- 包名 / 命名空间：`alo.`、`com.alo.`、`org.alo`
- 配置键名：`alo.api.url`、`alo.version`、`alo.feature.flag`、其他以 `alo` 开头的配置项
- 构建依赖：`alo-core`、`alo-common`、`alo-service`、其他旧库 artifactId、groupId、module 名称
- 注释 / 文档：TODO、FIXME 中提到旧库、旧项目代号、旧链接
- 结构残留：日志标记 `[ALO]`、API 路径 `/alo/api/...`、数据库表名前缀 `alo_`
- 旧库特有异常类型、工具类、客户端类、服务名

## 团队成员与职责

| 成员角色 | 核心职责 | 必备技能 |
|---|---|---|
| 重构分析负责人 | 统筹扫描范围、判断遗留风险、汇总报告 | 架构理解、迁移评估、风险分级 |
| 旧库特征识别专家 | 识别 `alo-`、`alo.`、`[ALO]`、旧 API、旧表名前缀等模式 | 正则、命名规范分析、代码审计 |
| 静态代码扫描工程师 | 扫描源码、配置、脚本、文档、锁文件 | `rg` / AST 搜索、依赖分析、文件分类 |
| 构建与依赖分析工程师 | 检查 `package.json`、`pom.xml`、Gradle、Go、lock 文件等 | 前后端构建系统、包管理、依赖溯源 |
| DevOps/CI 分析工程师 | 检查 Docker、CI/CD、部署路径、环境变量 | GitHub Actions、Docker、环境配置 |
| 文档与注释审计员 | 检查 README、docs、TODO、FIXME、旧链接 | 文档审计、注释语义识别 |
| 报告与验证工程师 | 生成迁移报告、风险等级、整改建议 | 报告结构化、证据定位、复查流程 |

## 风险分级规则

| 等级 | 定义 | 处理要求 |
|---|---|---|
| P0 | 会导致新库仍依赖旧库 | 必须立即处理 |
| P1 | 运行时、构建时、部署时可能受旧库影响 | 优先处理，需测试验证 |
| P2 | 文档、注释、命名残留，影响工程整洁度 | 建议清理，可随迁移批次处理 |
| P3 | 低风险历史痕迹 | 可延后处理或归档说明 |

## 命中类型分类

报告至少使用以下命中类型：

- 命名前缀残留
- 包名 / 命名空间残留
- 配置项残留
- 构建依赖残留
- 注释 / 文档残留
- API / 路由残留
- 数据库表名残留
- 日志标记残留
- 其他疑似旧库风格残留

## 扫描流程

1. 静态代码扫描工程师建立有效文件清单，确认排除目录和锁文件覆盖情况。
2. 旧库特征识别专家按关键词、正则和路径名三类线索扫描旧库痕迹。
3. 构建与依赖分析工程师检查构建文件和锁文件，识别旧 artifact、groupId、module、包名依赖。
4. DevOps/CI 分析工程师检查 Docker、CI/CD、部署脚本、环境变量和发布路径。
5. 文档与注释审计员检查 README、docs、OpenSpec、TODO、FIXME 和历史说明。
6. 重构分析负责人合并证据，按 P0-P3 进行风险分级。
7. 报告与验证工程师输出扫描报告、清理方案和复查建议。

## 报告格式

旧库残留扫描报告必须按以下表格输出：

| 风险等级 | 文件路径 | 行号 | 命中内容 | 命中类型 | 说明 | 建议处理方式 |
|---|---|---:|---|---|---|---|

每条清理建议必须说明：

- 是否建议立即修改。
- 修改方式。
- 是否需要测试。
- 是否可能影响功能。
- 是否需要人工确认。

## 清理方案输出要求

扫描完成后，需要输出 2 到 3 套清理方案。每套方案包含：

- 适用场景
- 清理范围
- 优点
- 缺点
- 风险
- 工作量评估
- 推荐程度

## SPAC 执行模式

用户选择清理方案后，按以下模式执行：

- S：梳理现状，列出所有旧库残留证据。
- P：制定迁移与清理策略。
- A：执行清理，包括重命名、配置替换、依赖移除、文档更新。
- C：校验闭环，重新扫描并输出验证报告。

## 工作准则

- 每个结论必须有证据位置，不凭印象判断。
- 每个修改建议必须区分“可自动处理”和“需要人工确认”。
- 对兼容迁移逻辑、历史备份和用户数据路径要谨慎处理，避免误删有效迁移能力。
- 对测试夹具、归档文档和历史变更记录要单独标注，不与运行时依赖混为一类。
- 在用户未确认清理方案前，不执行重命名、删除或替换。
