## Context / 背景

当前移动端记录页已经具备灵感捕捉和每日反思保存能力，且二者分别写入 `inspirations` 与 `reflections`。问题不在数据路径，而在交互表达：步骤节点看起来可点击但不是明确的 Stepper，外侧装饰线又延伸到历史列表，导致流程和历史混成一个视觉区域。

同时，`inboxItems` 与 `BACKLOG` tasks 是两类不同数据：前者是未澄清想法，后者是已确认但未安排日期的任务。现有文案多处把二者都叫「收纳箱」，用户会合理怀疑功能重复。

## Goals / Non-Goals / 目标与非目标

**Goals / 目标:**

- 将灵感和反思流程改为同一套可点击 Stepper。
- 让 Stepper 支持可变步骤数量，避免只适配灵感 4 步。
- 当前步骤切换时只展示该步骤的输入或确认区域。
- 将最近灵感/最近反思历史列表移出流程线范围。
- 用文案区分「待澄清」和「待安排任务」。
- 保持现有保存目标、同步载荷和移动端四主入口不变。

**Non-Goals / 非目标:**

- 不新增 `CaptureItem`、`EvidenceRecord` 或 backlog 专属模型。
- 不改变 `InboxItem`、`Task`、`Reflection`、`Inspiration` 字段结构。
- 不重做整个移动端视觉系统。
- 不把移动端记录页扩展成桌面完整澄清器。
- 不改变 V3 同步、Android 私有存储或 Electron IPC。

## Decisions / 设计决策

1. **抽出本地 Stepper 渲染单元，但不拆数据模型。**
   - 选择：在 `MobileCaptureHub.tsx` 内定义可复用 step metadata 和 stepper 渲染函数。
   - 理由：当前需求局限在记录页，先避免新增跨页面抽象。

2. **步骤是按钮，不是装饰节点。**
   - 选择：每个步骤使用 `button`，提供 `aria-current`、完成状态和禁用/可用语义。
   - 理由：用户期望「一个一个点」，交互必须匹配视觉。

3. **灵感和反思共用 Stepper，步骤数量可变。**
   - 选择：灵感步骤仍为想法、来源、标签、确认；反思步骤来自每日反思模板的核心问题并追加确认。
   - 理由：反思流程天然比灵感更长，不能硬编码 4 个点。

4. **历史区独立于流程区。**
   - 选择：流程卡只包含 header、说明、stepper、当前输入/确认和保存/下一步操作；最近列表作为后续 section。
   - 理由：流程线不能穿过历史记录，否则用户会误以为历史也是流程的一部分。

5. **命名只改用户可见边界，不改数据。**
   - 选择：`inboxItems` 显示为待澄清/捕捉，`BACKLOG` tasks 显示为待安排任务。
   - 理由：底层复用是正确的，问题是心智命名不清。

## Risks / Trade-offs / 风险与取舍

- Stepper 可点击后，用户可能跳到确认步骤但必填内容未填 → 确认区需要显示「先补齐必填项」并禁用保存。
- 反思模板问题数量可能变化 → Stepper 必须由模板问题派生，不能假设固定 5 或 6 步。
- 文案改名会影响已有测试快照或结构测试 → 新增结构测试锁定新的产品边界。
- 不拆出新组件会让 `MobileCaptureHub.tsx` 继续偏大 → 本轮保持局部修改，后续如文件继续膨胀再单独拆组件。
