## Context / 背景

系统页当前承载关于与更新、数据备份、COS 数据同步三类能力。COS/V3 同步已经展示较多底层字段，但用户仍缺少一个聚合判断：“本地数据是否存在、同步是否启用、是否有冲突、是否有待上传、本机数据在哪里”。同时，产品评审文档要求把 Forge-OS 的使用节奏产品化，避免用户把周看板、反思、周复盘、月度 OKR 看成彼此孤立的功能。

本次实现第一阶段：系统信任与使用节奏。实现必须保持只读展示，不改变持久化字段、同步流程、Electron IPC、Android 私有存储或 V3 合并逻辑。

## Goals / Non-Goals / 目标与非目标

**Goals / 目标:**

- 在系统页增加数据健康摘要，聚合本地数据、同步状态、冲突、待上传、存储路径和最近同步信息。
- 在系统页增加折叠式使用指南，说明早上、白天、晚上、周末、月底的使用仪式。
- 在 README 增加对应使用节奏说明。
- 使用结构测试锁定入口、文案和只读边界。

**Non-Goals / 非目标:**

- 不新增 `ritual`、`health` 或 `EvidenceRecord` 等持久化实体。
- 不修改 `AppState`、`SyncStatus`、V3 envelope、V3 runner 或 merge engine。
- 不增加诊断修复、重置 baseline、清理数据或自动同步触发动作。
- 不实现桌面收集箱、模块阶段分组或成长证据档案；这些留给后续 change。

## Decisions / 设计决策

1. **健康摘要使用派生视图，而不是新 store 字段。**
   - 选择：从 `useAppStore` 中已有任务、反思、灵感、同步配置和同步状态派生展示。
   - 理由：数据健康是当前状态的读取结果，不需要持久化；这样不会扩大 V3 同步契约。
   - 未选择：新增 `dataHealth` slice。该方案会增加同步字段和迁移成本，不符合第一阶段边界。

2. **使用指南放在系统页折叠组件，并同步 README。**
   - 选择：系统页提供产品内可见节奏，README 提供外部说明。
   - 理由：用户既能在应用内看到，也能在项目基板文档中找到一致描述。
   - 未选择：新增独立“仪式”页面或完成率统计。当前只需要引导，不需要新模型。

3. **组件放在 `src/features/system`。**
   - 选择：新增 `DataHealthPanel.tsx` 和 `UsageRitualPanel.tsx`，由 `System.tsx` 引入。
   - 理由：系统页维护类能力集中在同一目录，符合现有结构。
   - 未选择：塞进 `SyncPanel.tsx` 或 `DataBackupPanel.tsx`。这会让同步/备份组件继续膨胀，边界不清。

4. **测试优先使用结构测试。**
   - 选择：新增或扩展 `tests/systemTrustStructure.test.ts`，通过源码结构和文案断言验证入口和只读边界。
   - 理由：当前项目已有大量结构测试；该功能以展示为主，结构测试能低成本锁定回归。
   - 未选择：引入 React 测试库。项目当前没有该依赖，引入会扩大工具链。

## Risks / Trade-offs / 风险与取舍

- 健康摘要过度承诺同步安全 → 使用“只读摘要”和保守文案，避免显示为自动修复或诊断动作。
- 系统页信息密度增加 → 使用独立卡片和短行摘要，指南默认折叠，避免挤压 COS 配置。
- README 与产品内文案漂移 → 将相同五段仪式写入 `systemCopy` 和 README，并用测试覆盖关键短语。
- 结构测试无法验证真实渲染 → 后续 UI QA 可补浏览器截图；本次先保证功能入口、文案和只读边界。
