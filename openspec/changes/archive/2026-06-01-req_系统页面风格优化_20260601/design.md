## Context / 背景

当前应用由 `App.tsx` 控制页面状态和顶部导航，已有页面包括周看板、反思库和系统页。前一版实现新增了独立的 `SystemOrbit` 页面和 `[◇ 新系统]` 入口，但这不符合最新需求：用户需要的是风格切换，而不是新页面跳转。

本需求假设：风格切换只作用于整体视觉外壳、导航表现、颜色 token、卡片质感和交互反馈；页面功能仍由现有 `Dashboard`、`Reflection`、`System` 等组件承载，数据源和业务逻辑保持同一套。

## Goals / Non-Goals / 目标与非目标

**Goals / 目标:**

- 移除 `[◇ 新系统]` 跳转入口，改为 `[◇ 风格切换]`。
- 点击 `[◇ 风格切换]` 后，在原风格和 orbit-general 参考风格之间切换。
- 新风格下仍使用周看板、反思库、系统页、模块管理和主题切换这些原有功能入口。
- 原页面组件、数据源和业务逻辑保持同一套，不因为风格切换而分叉。
- 窄屏下导航和页面内容不重叠、不被裁切。

**Non-Goals / 非目标:**

- 不新增第二套路由或第二套业务页面。
- 不保留 `[◇ 新系统]` 页面入口。
- 不重写任务、反思、系统、备份、更新检查、模块管理或主题切换逻辑。
- 不引入新依赖或新的持久化字段。
- 不把参考页面中的业务内容、第三方连接器或脚本搬进应用。

## Decisions / 设计决策

- 使用应用级视觉模式状态，而不是新增页面状态。
  - 理由：风格切换是视觉外壳切换，不应改变当前功能路由；保留 `dashboard/reflection/system` 页面状态可以避免业务逻辑分叉。
  - 备选方案：继续维护 `systemOrbit` 独立页面；不采用，因为它会形成跳转入口，并且容易复制系统页业务逻辑。

- 用同一套页面组件承载功能。
  - 理由：`Dashboard`、`Reflection`、`System` 已经接入当前 store 和业务行为；在外层切换样式变量和导航表现即可共享数据源。
  - 备选方案：为每个页面复制一份 orbit 版本；不采用，因为会增加维护成本并造成业务逻辑重复。

- 将 orbit-general 转译为局部视觉 token 和外壳样式。
  - 理由：参考页是静态 HTML，直接复制会破坏现有 React 组件边界；局部 token 能让现有组件在新风格下换肤。
  - 备选方案：全局替换设计 token；不采用，因为风格需要可切回原样。

## Risks / Trade-offs / 风险与取舍

- 现有组件内联样式较多，新风格无法做到完全等同参考页：通过局部 CSS 变量、导航样式和卡片外观尽量接近 orbit-general。
- 新增风格状态可能被误认为业务状态：该状态不进入 store，不持久化，不参与数据逻辑。
- 导航入口变多后窄屏可能拥挤：导航按钮区域需要保持单行并允许横向滚动。
- 只靠 hover 表达交互会影响键盘和触屏用户：新风格下需要保留 focus-visible 和 active 反馈。

## Migration Plan / 迁移计划

1. 移除 `systemOrbit` 页面状态和 `[◇ 新系统]` 入口。
2. 移除不再使用的独立系统风格页组件。
3. 在 `App.tsx` 增加本地视觉模式状态，并新增 `[◇ 风格切换]` 按钮。
4. 在新风格下通过局部 CSS 变量和外壳类名改变导航、背景、卡片和按钮视觉。
5. 验证原风格和新风格都能访问周看板、反思库和系统页，且数据与操作行为一致。

## Open Questions / 开放问题

- 风格切换是否需要在未来持久化为用户偏好；当前需求不持久化，避免引入数据结构变化。
