## Context / 背景

Forge-OS 手机端已经形成「今日、推进、记录、系统」四个底部导航入口，并且 Android WebView、本地优先存储和 V3 COS 同步已经可用。当前问题集中在移动端闭环细节：有些动作有入口但缺少移动端去处，有些状态信息过度暴露调试字段，有些记录反馈混合导致语义不清。

本设计把这批问题作为一个移动端日常闭环优化包处理。核心约束是：所有 UI 和交互调整都必须保留现有业务能力，尤其是任务本地写入、灵感保存、每日反思保存、V3 同步和 Android 离线可用性。

## Goals / Non-Goals / 目标与非目标

**Goals / 目标:**

- 让手机端「推进」页具备完整收纳箱闭环：收纳、查看、新增、编辑、删除、安排到日期。
- 让手机端「记录」页的信息架构更清晰：灵感和反思分别拥有输入、保存反馈和最近列表。
- 让灵感捕捉流程统一为纵向递进节点，保留一步一步展开的移动端交互体验。
- 让写灵感和写反思的展开态可以在当前页面收起。
- 让同步面板默认展示当前有效 V3 状态，避免旧基线和 V3 基线并列误导。
- 让移动端风格支持本机独立设置，默认仍跟随桌面。
- 保留所有现有移动端核心功能，不回退桌面端行为。

**Non-Goals / 非目标:**

- 不重写任务、灵感、反思业务数据模型。
- 不改造 COS V3 合并引擎。
- 不新增第五个移动端底部导航。
- 不把收纳箱绑定到某一周。
- 不做桌面端视觉大改版。
- 不引入新的移动端测试框架或 Android 原生 UI 重写。

## Decisions / 设计决策

### Decision 1: 收纳箱放在推进页周日下方，作为跨周公共池

选择：在 `MobileWeekProgress` 的周一到周日节点之后展示收纳箱，收纳箱展示 `date === 'BACKLOG'` 的任务。

原因：收纳是任务推进动作的后续状态，应在「推进」模块内完成闭环。放在周日下方符合用户要求，也避免新增底部导航或把收纳塞进系统页造成低频入口。

备选方案：

- 放在今日页：更靠近今日承诺，但会把长期未安排任务混入今日语境。
- 放在系统页：实现简单，但用户刚收纳后很难找到。
- 新增底部导航：入口强，但为一个子任务池占用主导航过重。

### Decision 2: 收纳任务继续复用现有 Task，并用 BACKLOG 表示无日期

选择：继续使用现有 `Task`，未安排任务使用现有 `date === 'BACKLOG'` 语义。新增、编辑、删除和安排都走现有 task slice 能力。

原因：这样不会引入 Android 专属任务模型，也不会破坏 V3 同步实体结构。安排到具体日期时，可以复用 `moveTask(taskId, targetDate, order)`。

备选方案：

- 新增 backlog 实体：表达更干净，但会扩大数据迁移和同步风险。
- 给 Task 增加 `isBacklog` 字段：语义更直接，但会改变业务模型，并与现有 `BACKLOG` 约定重复。

### Decision 3: 收纳安排使用底部面板

选择：点击收纳任务的「安排」后打开底部面板。面板优先展示当前查看周的 7 天快捷按钮，并提供「选择其他日期」。

原因：手机端日期安排需要更大的触控区域和更明确的上下文。底部面板能承载 7 天快捷按钮，也能扩展到其他日期，而不会让每个任务卡片变得更拥挤。

备选方案：

- 任务卡直接放多个日期按钮：快捷但列表噪音大。
- 只使用日期选择器：通用但高频安排本周任务时成本太高。

### Decision 4: 任务操作区减轻视觉重量但保留触控热区

选择：把当前大圆按钮改为更紧凑的 action row、文字按钮或图标+文本按钮。视觉尺寸变轻，但点击热区仍满足移动端可用性。

原因：当前按钮过大导致列表密度下降。移动端任务管理需要扫读任务内容，操作按钮应服务任务，而不是抢视觉主导。

### Decision 5: 记录页取消混合记录流

选择：取消底部统一「记录流 / 最近保存」，改为灵感卡片内部「最近灵感」和反思卡片内部「最近反思」。

原因：灵感和反思是不同认知成本、不同归档目的的记录。混合流降低了用户对保存去向的判断，也让两个卡片的闭环不完整。

### Decision 6: 灵感流程改为纵向递进节点

选择：保留 `想法 -> 来源 -> 标签 -> 确认` 的一步一步体验，但取消横向 stepper，统一成纵向节点。当前步骤展开，已完成步骤显示摘要，未开始步骤弱化。

原因：移动端纵向阅读和输入更自然，也与当前卡片左侧节点视觉保持一致。横向步骤在窄屏上拥挤，并和竖线表达冲突。

### Decision 7: 展开态必须有本页收起路径

选择：写灵感和写反思都支持展开后收起。收起可以通过再次点击入口、明确「收起/取消」按钮，或两者结合实现。

原因：移动端空间有限，展开态不可逆会制造滚动负担。切换模块不应成为收起输入框的唯一方式。

### Decision 8: 移动端风格默认跟随桌面，可主动切为本机独立

选择：默认保持现有跟随桌面行为。用户开启「本机独立风格」后，移动端风格偏好只保存在当前设备或当前移动环境，不写入跨端同步风格。

原因：默认跟随桌面最少惊扰现有用户；本机独立风格满足手机端固定视觉的需求。该偏好必须与 V3 业务同步边界区分。

备选方案：

- 默认独立：更保护桌面，但会改变现有默认行为。
- 首次弹窗选择：更明确，但增加启动干扰。

### Decision 9: 同步主状态只展示当前有效 V3 信息

选择：当 V3 revision 存在时，旧基线不在主状态区显示。旧基线如需保留，进入调试或旧版快照信息区域。

原因：用户关心的是当前同步是否成功、是否有冲突、是否有待上传。旧基线和 V3 基线并列会让用户误以为存在两个日常同步来源。

## Risks / Trade-offs / 风险与取舍

- [按钮变小后触控变差] → 保留足够点击热区，只减轻视觉外形。
- [收纳箱放在周日下方导致低可见性] → 使用清晰标题、数量和空态；收纳动作后给出反馈，提示已进入收纳箱。
- [本机独立风格误入跨端同步] → 端侧偏好与同步 config 分离，测试覆盖默认跟随和独立两种状态。
- [拆分记录流后最近保存反馈丢失] → 分别在灵感卡片和反思卡片内提供最近列表和保存闪烁反馈。
- [隐藏旧基线影响排查] → 保留旧基线在次级调试区域，不放在主状态区。
- [实现中影响现有保存路径] → 规格和任务要求优先保持现有 `addInspiration`、`saveReflection`、`moveTask`、`deleteTask`、`runV3Sync` 路径。

## Migration Plan / 迁移计划

- UI 调整可以直接随前端版本发布。
- 收纳箱使用现有 `BACKLOG` 任务，不需要数据迁移。
- 记录页最近列表拆分只改变展示，不需要迁移灵感或反思。
- 风格本机独立偏好如需持久化，应新增本地偏好并默认关闭；旧数据缺字段时按跟随桌面处理。
- 如发布后发现本机独立风格异常，可以关闭该开关入口并回到默认跟随桌面，不影响业务数据。

## Open Questions / 未决问题

- 暂无阻塞性问题。具体按钮形态可以在实现阶段用移动端截图确认，但不得违反触控热区和功能不回退约束。
