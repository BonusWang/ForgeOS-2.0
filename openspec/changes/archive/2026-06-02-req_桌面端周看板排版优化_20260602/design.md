## Context / 背景

当前 `Dashboard` 页面先展示 `TaskBoard`，再以 61.8% / 38.2% 双栏承载今日进度、每日反思、时间块、原则、日历、娱乐、习惯、情绪和灵感库。电脑端可见空间足够，但模块层级混杂：今日执行状态在周看板之后，情绪追踪等今日相关模块又落在右侧后段，用户扫描成本较高。

本设计采用用户确认的版本 B：今日执行条、周计划区、辅助沉淀区。它是基础页面排版优化，不是风格模式分支；classic 和 orbit 必须共用同一结构。

## Goals / Non-Goals / 目标与非目标

**Goals / 目标:**

- 将桌面端周看板页面调整为今日执行条 → 周看板 → 辅助沉淀区的顺序。
- 复用现有业务组件，不复制模块逻辑，不新增业务状态。
- 让今日进度、每日反思、时间块、情绪追踪在电脑端形成清晰的今日执行入口。
- 让原则、日历、娱乐、习惯追踪、灵感库形成低优先级辅助区域。
- 保持周看板作为主要规划区，任务列和操作逻辑不变。

**Non-Goals / 非目标:**

- 不修改任务、反思、时间块、情绪、习惯、娱乐、灵感等 store 逻辑。
- 不新增持久化字段、数据迁移或 Electron IPC。
- 不重写 `TaskBoard` 的拖拽、添加、完成、周切换或周复盘逻辑。
- 不让风格切换改变基础排版。

## Decisions / 设计决策

- 在 `Dashboard` 层重排模块，而不是修改各业务组件内部。
  - 理由：需求是页面排版优化；业务组件已经封装各自状态和操作。只在页面层重排可以保持业务边界清晰。
  - 备选方案：改造每个模块内部结构；不采用，因为会扩大改动面并触及业务交互细节。

- 使用三个语义布局区：`dashboard-today-strip`、`dashboard-board-section`、`dashboard-support-grid`。
  - 理由：分区命名能让后续维护者明确模块优先级，也方便在 CSS 中按桌面/窄屏调整布局。
  - 备选方案：继续使用原双栏布局；不采用，因为今日相关模块分散，无法形成打开页面后的执行入口。

- 今日执行条使用响应式网格。
  - 理由：电脑端可以横向并列显示，窄屏时自然降为单列或少列，不改变模块集合。
  - 备选方案：固定四列；不采用，因为较窄桌面和移动端容易挤压内容。

- 辅助沉淀区使用自适应网格。
  - 理由：辅助模块高度差异较大，自适应网格比固定左右栏更容易保持整齐和可扫描。
  - 备选方案：三栏固定布局；不采用，因为会在 1366px 等宽度下过于拥挤。

## Risks / Trade-offs / 风险与取舍

- 今日执行条中模块高度不一致 → 使用 `align-items: stretch` 和统一容器间距缓解，不强行压缩业务组件内容。
- 情绪追踪模块内容较长，可能拉高今日执行条 → 接受该模块自然高度，避免裁切表单或改变业务交互。
- 周看板宽列较多 → 保留 `TaskBoard` 内部横向滚动，避免页面级横向溢出。
- 风格切换可能再次引入结构分支 → 只在 `Dashboard` 基础结构中实现排版，风格差异由外层 token/CSS 处理。
