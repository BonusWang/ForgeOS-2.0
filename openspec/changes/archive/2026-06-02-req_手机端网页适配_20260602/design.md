## Context / 背景

Forge-OS 是 React + Vite + Electron 项目，但开发页也会作为普通网页在浏览器中运行。`index.html` 已包含移动端 viewport meta，当前问题主要集中在应用层布局：顶部导航、页面主体、任务看板、双列页面、弹层和表单在手机宽度下仍可能拥挤或横向溢出。

现有代码已经有一部分响应式规则，例如 `src/index.css` 和 `src/App.tsx` 中的 `max-width: 767px` 媒体查询，以及看板区域的横向滚动。后续实现应延续这些入口，避免为移动端复制页面或拆出第二套业务组件。

## Goals / Non-Goals / 目标与非目标

**Goals / 目标:**

- 让周看板、周复盘、反思库、系统页和模块选择器在常见手机浏览器宽度下可读、可点、可输入。
- 通过响应式 CSS 和少量 class hook 调整布局，保留现有 React 页面状态、store、Electron IPC 和持久化格式。
- 保持桌面端现有布局和 orbit / classic 两套视觉风格的基本表现不回退。
- 为后续继续细化移动端体验留下 OpenSpec 可扩展空间。

**Non-Goals / 非目标:**

- 不实现独立移动 App、PWA 安装、离线同步或跨设备同步。
- 不重做信息架构，不把顶部导航替换为完整移动端抽屉菜单。
- 不修改任务、反思、OKR、模块配置等数据模型。
- 不引入新的 UI 框架或响应式依赖。

## Decisions / 设计决策

### Decision 1: 以 CSS-first 响应式改造为主

优先在 `src/index.css` 和 `src/App.tsx` 已有媒体查询中补齐手机端规则，必要时给现有组件增加稳定 className。这样可以集中处理通用问题，并减少对业务组件逻辑的触碰。

备选方案是为手机端创建独立页面组件或使用 JS 检测 viewport 后切换渲染。该方案短期可控，但会引入两套布局路径，后续维护成本更高，也更容易让桌面端和移动端行为分叉。

### Decision 2: 顶部导航采用可换行和横向滚动的轻量适配

手机端导航应保留品牌、核心页面入口、模块入口、主题入口和风格切换入口。导航容器在窄屏下允许分成品牌行和操作行，操作行可横向滚动，按钮保持足够触控面积并避免文字裁切。

暂不引入汉堡菜单。原因是第一版目标是网页可用性，不是重做移动端导航模型；保留可见入口更利于发现和测试。若后续手机端入口过多，再通过同一 change 扩展为折叠菜单。

### Decision 3: 页面主体按“单列优先、局部横滚”处理

Dashboard、Reflection、WeeklyReview、System 等页面在手机端应优先降为单列，减少左右并排布局。对于任务看板这种天然多列信息，保留局部横向滚动，并用稳定列宽保证拖拽和浏览不破坏整体页面宽度。

备选方案是把周看板改为手机端按天折叠列表。该方案体验可能更好，但会改变任务看板的信息结构和交互模型，适合后续单独迭代，不放入第一版适配。

### Decision 4: 弹层和表单使用视口约束

ModulePicker、ReflectionDetailModal 以及反思、周复盘等输入区域需要在手机端使用 `width: calc(100vw - safe margin)`、`max-height` / `max-width`、内部滚动和更舒适的 padding，避免弹层超出屏幕或输入框过窄。

按钮和输入控件应在手机端保证可触控尺寸，但不改变保存、编辑、关闭等业务行为。

### Decision 5: 以真实视口验证，而不是只看构建通过

实现完成后至少验证桌面宽度和常见手机宽度。建议覆盖 360px、390px、430px 和 768px 左右视口，重点检查导航、看板横滚、周复盘输入、反思详情弹层、系统页备份面板和模块选择器。

## Risks / Trade-offs / 风险与取舍

- 手机端保留任务看板横向滚动 → 看全周任务需要左右滑动；第一版通过稳定列宽和局部滚动控制可用性，后续可再设计按天折叠视图。
- 大量布局写在内联 style 中 → 纯 CSS 覆盖会受到限制；必要时只增加 className 或少量样式 props，不做无关重构。
- 导航入口较多 → 手机端仍可能需要横向滚动；第一版优先保证入口不丢失，后续再评估是否引入折叠菜单。
- orbit 与 classic 两套视觉共存 → 同一媒体查询可能影响两套风格；实现时需要分别检查 `body[data-alo-visual-style]` 和 `.orbit-style-shell` 下的表现。

## Migration Plan / 迁移计划

1. 补齐通用移动端 CSS 和必要 className，不迁移数据。
2. 分页处理导航、主页面布局、看板、弹层和表单，保持每一步可独立验证。
3. 运行静态检查和构建，随后用移动视口人工或浏览器自动化检查核心页面。
4. 如需回滚，移除本次新增的移动端 CSS 和 className 即可，不涉及数据回滚。

## Open Questions / 未决问题

- 手机端顶部导航后续是否需要折叠菜单，或继续保留当前可见入口模型？
