## Why / 背景

当前 COS 同步仍以整份快照和 MVP 级实体合并为主，遇到“Android 新增任务、浏览器端也有其他改动”时容易让用户在“采用云端/保留本地”之间来回操作，甚至需要手工恢复快照。现在浏览器端数据已确认为唯一最新真源、已导出本地备份、其他端已暂停使用、COS 前缀已清理，适合从干净基线实现 V3 同步引擎。

## What Changes / 变更内容

- 新增 V3 同步主对象 `alo-data.entities.v3.json`，日常同步优先使用 V3。
- 业务数据结构保持不变；同步引擎单独维护本地同步缓存、云端同步元数据、删除 tombstones 和冲突列表。
- 用浏览器端当前本地数据初始化第一份 V3 云端基线，不自动读取或迁移旧云端 v1/v2 主同步对象。
- 实现完整三方合并：`base + local + remote`，自动合并不同实体和不同字段改动。
- 实现删除 tombstone，避免一端删除的数据被另一端旧数据复活。
- 将真冲突收敛到字段级或实体级冲突，并持久化冲突记录，避免反复提示同一冲突。
- 预留多用户隔离：V3 同步对象和 envelope 必须带同步命名空间，当前以 objectPrefix/profileId 区分，未来可映射 userId。
- 保留现有 v1 快照和历史恢复作为手工备份/灾难恢复能力，但不再参与 V3 日常同步决策。

## Capabilities / 能力范围

### New Capabilities / 新增能力

- 无

### Modified Capabilities / 修改能力

- `腾讯云COS数据同步`: 将日常跨端同步从整份快照升级为 V3 实体级同步引擎，要求支持干净初始化、三方合并、删除 tombstone、持久化冲突记录、多用户命名空间隔离和本地优先失败保护。

## Impact / 影响范围

- 影响同步核心代码：COS object key、同步客户端、实体快照转换、三方合并引擎、启动/自动/手动同步 hook。
- 影响同步状态展示：系统页需要展示 V3 基线、合并数量、冲突数量、初始化状态和错误信息。
- 影响同步路径约定：V3 主对象必须落在当前用户/资料的同步命名空间下，避免不同用户写入同一对象。
- 影响测试：新增 V3 快照、合并、删除 tombstone、冲突持久化、干净初始化和 hook wiring 测试。
- 不修改业务实体字段，不新增用户功能模块，不改变当前任务、反思、心情、灵感等业务 UI。
