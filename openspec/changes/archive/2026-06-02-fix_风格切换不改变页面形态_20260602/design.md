# Style Switch Shape Invariance Fix Design / 风格切换不改变页面形态修复设计

## Context / 背景

风格切换的目标是让用户在同一套功能页面上更换视觉皮肤。此前 orbit 风格相关要求允许新增页头、KPI 或状态卡片，容易让切换后的页面看起来像另一套业务结构，和“同一页面换视觉表现”的产品理解冲突。

## Goals / Non-Goals / 目标与非目标

目标：

- 风格切换只改变视觉表现，不改变页面业务结构。
- classic、orbit 等风格共享同一导航、模块集合、模块顺序和业务布局。
- 周看板、反思库、周复盘、系统页在不同风格下保持可识别的同一页面形态。

非目标：

- 不新增业务模块。
- 不调整 store、持久化格式、Electron IPC 或数据迁移逻辑。
- 不把风格切换扩展为布局模式切换。

## Decisions / 设计决策

1. 将风格状态限制为外层视觉类、设计 token、背景、边框、表面质感和交互态的输入。
2. 页面组件不得因为当前风格不同而分支渲染不同业务模块。
3. 导航文案、logo 显隐、周看板列数量、模块显隐和业务 DOM 集合应在风格切换前后保持一致。
4. orbit 风格不再通过新增 KPI、页头或状态卡片来表达差异，只通过视觉层次和局部样式体现风格。

## Risks / Trade-offs / 风险与取舍

- 风格之间的差异会更克制，但用户不会因为切换皮肤而丢失页面位置感。
- 后续如果需要不同布局，应作为独立布局设置或新需求提出，而不是复用风格切换开关。
