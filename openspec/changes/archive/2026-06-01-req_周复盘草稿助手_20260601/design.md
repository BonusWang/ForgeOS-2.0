# Weekly Review Draft Assistant Design / 周复盘草稿助手设计

## Context

> 中文：背景

Forge 的长期产品原则是“先假模假样，最后像模像样”：先用真实数据和规则模板跑通产品闭环，再逐步替换背后的智能能力。

周复盘草稿助手是这个原则的第一项落地需求。它不是 AI 助手，而是具备 AI 化体验的规则草稿器。

## Decisions

> 中文：设计决策

1. 第一版不接真实 AI，不调用外部模型，不要求 API Key。
2. 入口放在周执行页面，保存结果归入反思沉淀。
3. 采用轻量面板或弹窗，不先做独立页面。
4. 草稿结构固定为三段：事实报告、教练追问、下周调整建议。
5. 数据来源只读：本周任务、当前月 OKR/目标、能力数据、本周反思。
6. 保存时必须生成独立的周复盘记录，不能覆盖同一天的每日反思。
7. 周复盘记录需要包含周周期信息，为未来月度成长报告提供证据链。

## User Flow

> 中文：用户流程

1. 用户在周执行页面点击“周复盘草稿”入口。
2. 系统读取当前周的数据。
3. 系统打开轻量面板，展示三段式草稿。
4. 用户可以编辑草稿。
5. 用户点击保存后，草稿作为“周复盘”进入反思沉淀。
6. 用户取消或关闭面板时，不写入任何数据。

## Draft Structure

> 中文：草稿结构

事实报告：

- 本周完成任务数量和未完成任务数量。
- 与当前月目标/OKR 相关的行动证据。
- 有能力关联的任务和能力训练证据。
- 本周已有反思数量和可见主题。

教练追问：

- 哪个目标缺少行动证据？
- 哪个能力说重要但本周没有训练？
- 哪类任务反复拖延或留到收纳箱？
- 有没有把“准备”伪装成“推进”？

下周调整建议：

- 下周保留什么。
- 停止什么。
- 哪个目标需要拆小。
- 哪个能力需要明确训练动作。

## Data Design

> 中文：数据设计

当前反思存储按 `date` 去重保存，因此实现时必须避免周复盘覆盖当天每日反思。

推荐方向：

- 为反思记录增加可选类型字段，例如 `kind: 'daily' | 'weeklyReview'`。
- 为周复盘增加周期字段，例如 `periodStart`、`periodEnd`。
- 保存周复盘时按记录 `id` 或 `kind + periodStart` 区分，而不是只按 `date` 覆盖。

如果第一版选择更小改动，也必须保证：

- 用户已有每日反思不被覆盖。
- 周复盘可以在反思库中被识别和筛选。
- 后续月度成长报告能读取到周复盘周期和内容。

## Risks

> 中文：风险

- 如果继续按 `date` 保存反思，周复盘可能覆盖每日反思。
- 规则模板生成的内容如果太像 AI 断言，可能显得装腔；文案应保持“草稿”和“建议”的语气。
- 草稿过长会增加使用负担，第一版要克制。
- 如果保存后无法在反思库中识别“周复盘”，未来月报证据链会变弱。

## Validation

> 中文：验证方式

- OpenSpec 校验通过。
- 周执行页面能打开周复盘草稿面板。
- 无 AI/API Key 时功能可用。
- 保存周复盘不会覆盖当天每日反思。
- 反思库能看到或筛选周复盘记录。
- 源任务、OKR、能力和反思数据不会被草稿生成动作修改。
